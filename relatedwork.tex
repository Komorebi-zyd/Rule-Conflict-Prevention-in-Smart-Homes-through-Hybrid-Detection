\section{Related Work}

Prior work on rule conflict detection for smart home systems spans security-policy checking, formal verification, interaction-pattern analysis, and hybrid detection. Huang et al.~\cite{huang2023survey} note that the problem definition, detection target, and conflict taxonomy of smart home rule conflict detection are still not fully unified.

\textbf{Policy/property checking.} Soteria~\cite{celik2018soteria} extracts an intermediate representation and uses model checking to detect property violations in individual apps and app interactions, but it does not model physical-channel effects. IoTGuard~\cite{celik2019iotguard} enforces predefined safety/security policies at runtime by blocking unsafe actions before execution; however, its protection is mainly expressed at the action layer and cannot capture conflicts that arise via physical-channel propagation (e.g., an action changes an environmental variable that later triggers another rule). SafeChain~\cite{hsu2019safechain} models trigger--action programs as state machines to detect attack chains and reports repair suggestions, but its “dynamic” part primarily reacts to rule updates rather than confirming whether a candidate rule conflict is activated when rules are triggered.

\textbf{Physical-channel-aware analysis.} iRuler~\cite{wang2019charting} and IoTIE~\cite{chen2019multi} analyze interactions through physical channels by mapping rules/actions to environmental variables, which helps expose hidden interactions beyond shared devices. However, their zone awareness depends on deployment-specific input (or is limited), which may lead to false positives when physical-channel propagation across zones is over-approximated. IoTSafe~\cite{ding2021iotsafe} further combines static analysis with dynamic monitoring to validate real physical interactions and builds physical models for prediction, but it introduces substantial deployment/testing effort and can be sensitive to environmental disturbances. IoTCom~\cite{alhanahnah2022iotcom} identifies interaction threats across cyber and physical channels via static analysis and bounded model checking, yet it lacks explicit zone-aware propagation modeling and thus may over-report cross-zone interactions.

\textbf{Hybrid monitoring and handling.} IoTMediator~\cite{chi2023detecting} combines static pre-screening with dynamic monitoring and supports customized handling of detected threats, but it does not explicitly model physical-channel effects or zone-dependent propagation. In contrast, our framework models zone-aware physical-channel propagation, uses a user-defined Entity Safety Configuration to filter general interactions into candidate rule conflicts, and applies runtime assertion verification to confirm whether a candidate rule conflict edge is activated under the current system state. This design reduces false positives while keeping the runtime overhead at the millisecond level.
