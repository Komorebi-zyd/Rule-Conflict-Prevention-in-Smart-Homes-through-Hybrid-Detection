\section{Related Work}

Prior work on rule conflict detection for smart home systems spans policy/property checking, interaction-pattern analysis, and hybrid detection. Huang et al.~\cite{huang2023survey} note that the problem definition, detection target, and conflict taxonomy are still not fully unified. We summarize representative approaches by whether they distinguish rule interaction from candidate rule conflicts (DIC) and whether they incorporate user-specific safety preferences with reasonable user effort.

\textbf{Policy/property checking.} Soteria~\cite{celik2018soteria} extracts an intermediate representation and applies model checking to detect property violations in individual apps and app interactions. IoTGuard~\cite{celik2019iotguard} enforces predefined safety/security policies at runtime by blocking unsafe actions before execution. SafeChain~\cite{hsu2019safechain} and IoTSan~\cite{nguyen2018iotsan} verify security properties over trigger--action programs and report risky behaviors or attack chains. A common limitation is that these approaches rely on expert-crafted policies/properties (or templates), which can be difficult to generalize across diverse smart home deployments and often require substantial manual effort to adapt and maintain.

\textbf{Physical-channel-aware analysis.} iRuler~\cite{wang2019charting} and IoTIE~\cite{chen2019multi} analyze interactions through physical channels by mapping actions to environmental variables, which helps expose hidden interactions beyond shared devices. However, their support for zone-dependent propagation is limited or depends on deployment-specific input, which may over-approximate cross-zone effects and introduce false positives. IoTSafe~\cite{ding2021iotsafe} further combines static analysis with dynamic monitoring to validate real physical interactions and builds physical models for prediction, but it introduces substantial deployment/testing effort and can be sensitive to environmental disturbances. IoTCom~\cite{alhanahnah2022iotcom} detects interaction threats across cyber and physical channels via static analysis and bounded model checking, yet it does not explicitly model zone-dependent propagation.

\textbf{Interaction-pattern analysis and hybrid detection.} Pattern-based systems summarize representative interaction patterns (often from empirical observations) and flag automation rules that match these patterns~\cite{trimananda2020understanding,chi2020cross,chi2023detecting}. Such patterns are useful for discovering suspicious rule interaction, but a matched pattern may still require deployment-specific context to determine whether it should be treated as a candidate rule conflict. IoTMediator~\cite{chi2023detecting} combines static pre-screening with runtime dynamic monitoring and supports customized handling, yet its pattern-driven target is not directly aligned with DIC under user-specific safety preferences, which can lead to false positives in some deployments.

Compared with these approaches, our framework uses a user-defined Entity Safety Configuration as a lightweight preference baseline to filter rule interaction into candidate rule conflicts, and applies runtime assertion verification to confirm whether a candidate conflict edge is activated under the current system state. This design eliminates nine false positives reported by IoTMediator in our testbed while keeping the runtime overhead at the millisecond level.
